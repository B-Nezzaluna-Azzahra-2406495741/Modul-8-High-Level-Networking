# Modul 8 High Level Networking

## Reflection

1. What are the key differences between unary, server streaming, and bi-directional streaming RPC (Remote Procedure Call) methods, and in what scenarios would each be most suitable?  
   Perbedaan utamanya adalah jumlah message yang dipertukarkan dalam satu connection cycle. Unary gRPC memiliki pola single request dan single response, yang cocok untuk operasi sederhana seperti user authentication atau fetching satu baris data. Server streaming memungkinkan client mengirim satu request namun menerima continuous data stream dari server, cocok untuk skenario seperti real-time stock price updates atau pengiriman data besar dalam bentuk chunks. Sedangkan, bi-directional streaming memungkinkan kedua belah pihak mengirimkan sequence of messages secara independen dan simultan, menjadikannya pilihan terbaik untuk aplikasi real-time interactive seperti aplikasi chat.

2. What are the potential security considerations involved in implementing a gRPC service in Rust, particularly regarding authentication, authorization, and data encryption?  
   Keamanan dalam gRPC mencakup aspek authentication, authorization, dan data encryption untuk menjaga integritas sistem. gRPC memiliki built-in support untuk autentikasi dan load balancing yang membantu meningkatkan keamanan serta scalability sistem terdistribusi. Pada Rust, penggunaan metadata sering menjadi cara standar untuk menyertakan auth token pada setiap procedure call. Selain itu, enkripsi data biasanya ditangani melalui integrasi dengan TLS (Transport Layer Security) untuk memastikan bahwa komunikasi antar service tetap privat dan aman dari intersepsi pihak luar.

3. What are the potential challenges or issues that may arise when handling bidirectional streaming in Rust gRPC, especially in scenarios like chat applications?  
   Menangani bidirectional data streams, terutama dalam aplikasi chat, memunculkan tantangan terkait sinkronisasi dan connection state management. Tantangan utamanya mencakup error handling saat salah satu pihak mengalami unexpected disconnect, dimana memerlukan logika error handling yang kuat agar service tidak crash. Developer harus memastikan pesan diproses sesuai urutan (sequencing) dan menangani potensi deadlock atau race conditions saat banyak asynchronous tasks berjalan bersamaan untuk mengirim dan menerima message.

4. What are the advantages and disadvantages of using the tokio_stream::wrappers::ReceiverStream for streaming responses in Rust gRPC services?  
   Penggunaan ReceiverStream memberikan keuntungan berupa kemudahan integrasi antara MPSC channels milik Tokio dengan streaming interface milik Tonic/gRPC. Utility ini memungkinkan developer untuk mengubah channel receiver menjadi data stream yang bisa diproses oleh framework gRPC secara efisien. Namun, kekurangannya adalah developer harus mengelola channel capacity dan strategi backpressure secara manual. Jika kapasitas buffer tidak dikelola dengan benar, hal ini dapat menyebabkan memory bloat atau data loss jika salah satu pihak memproses data terlalu lambat.

5. In what ways could the Rust gRPC code be structured to facilitate code reuse and modularity, promoting maintainability and extensibility over time?  
   Struktur kode gRPC di Rust dapat ditingkatkan modularitasnya dengan memisahkan service module definitions menggunakan makro tonic::include_proto! ke dalam berkas atau modul terpisah. Mengikuti pola arsitektur seperti pembagian berkas menjadi controller, model, dan service (seperti pada standar Spring Boot) juga dapat diterapkan di Rust untuk memisahkan business logic dari networking logic. Penggunaan traits untuk implementasi service memungkinkan abstraction, sehingga komponen yang berbeda dapat menggunakan kembali definisi service yang sama dengan implementasi yang bervariasi.

6. In the MyPaymentService implementation, what additional steps might be necessary to handle more complex payment processing logic?  
   Implementasi MyPaymentService saat ini masih sangat basic karena hanya mengembalikan immediate success status. Untuk menangani logika yang lebih kompleks, diperlukan langkah tambahan seperti integrasi basis data untuk balance validation, komunikasi dengan third-party payment gateway, serta implementasi mekanisme rollback jika transaksi gagal di tengah jalan. Selain itu, diperlukan detailed logging dan penanganan berbagai gRPC error status codes untuk memberikan informasi yang akurat kepada client jika terjadi kegagalan sistem.

7. What impact does the adoption of gRPC as a communication protocol have on the overall architecture and design of distributed systems, particularly in terms of interoperability with other technologies and platforms?  
   Implementasi gRPC memberikan dampak signifikan, yaitu peningkatan efisiensi service-to-service communication yang lebih deskriptif karena menggunakan actual function calls alih-alih deskripsi HTTP yang sering ambigu. gRPC populer dalam arsitektur mikroservis karena kemampuannya menghubungkan service antar berbagai bahasa pemrograman secara seamless. Namun, adopsi ini juga memiliki batasan, terutama pada browser support yang masih rendah, sehingga penggunaannya lebih sering terbatas pada komunikasi internal di belakang API Gateway.

8. What are the advantages and disadvantages of using HTTP/2, the underlying protocol for gRPC, compared to HTTP/1.1 or HTTP/1.1 with WebSocket for REST APIs?  
   HTTP/2 memberikan keunggulan performa melalui fitur multiplexing yang memungkinkan pengiriman banyak request dan response secara concurrent dalam satu koneksi TCP, menghilangkan masalah Head-of-Line (HoL) blocking pada HTTP/1.1. Selain itu, fitur header compression (HPACK) dan binary framing secara drastis mengurangi data overhead dibandingkan format berbasis teks. Meskipun WebSocket menawarkan komunikasi full-duplex, gRPC di atas HTTP/2 memberikan struktur yang lebih terdefinisi dengan strict API contract dan efisiensi serialisasi yang lebih baik melalui Protocol Buffers.

9. How does the request-response model of REST APIs contrast with the bidirectional streaming capabilities of gRPC in terms of real-time communication and responsiveness?  
   Model request-response tradisional pada REST API bersifat searah (unidirectional) dan terbatas pada single cycle, di mana client harus selalu memulai komunikasi untuk mendapatkan data baru. Sedangkan, kemampuan bidirectional streaming pada gRPC memungkinkan komunikasi real-time yang lebih responsif karena server dapat melakukan push data ke client segera setelah tersedia tanpa menunggu explicit request. Hal ini mengurangi latency secara signifikan untuk aplikasi yang membutuhkan sinkronisasi data instan dengan high frequency.

10. What are the implications of the schema-based approach of gRPC, using Protocol Buffers, compared to the more flexible, schema-less nature of JSON in REST API payloads?  
    Schema-based approach pada gRPC menggunakan Protocol Buffers memberikan efisiensi performa tinggi karena data dikonversi ke dalam binary format yang jauh lebih ringan dan cepat diproses oleh CPU dibandingkan teks JSON. Schema yang ketat pada berkas .proto memastikan automatic message validation terhadap API contract, mengurangi resiko type mismatch antar service. Selain itu, JSON yang bersifat schema-less memberikan fleksibilitas lebih tinggi dan lebih readable, namun mengorbankan performa dan memerlukan langkah validasi manual tambahan pada sisi aplikasi.
